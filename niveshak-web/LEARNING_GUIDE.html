What is Niveshak.AI?
The Big Picture

The Problem Being Solved

Imagine you are Arun — an auto-rickshaw driver in Ranchi earning ₹2,500 a month. You want to save money for your daughter's education. But when you walk into a bank, the relationship manager uses terms like "risk appetite", "asset allocation", "ELSS", and "NPS Tier-1". You feel lost. You leave without investing anything.

This is the reality for 400 million working Indians. They want to invest but don't know how. Financial advisors are expensive or intimidating. Existing apps assume you already understand finance.

💡 The Insight

People don't need financial education first. They need a personalised recommendation first, in their own language, that tells them exactly what to do and why. Understanding follows action.

What Niveshak.AI Does

Niveshak.AI is a web application that works like a smart financial advisor in your pocket. Here is the user journey in plain language:

1

User answers 11 simple questions

Name, where they live, what work they do, how much they earn, what their financial goal is, how long they have, and how much risk they can stomach. All in Hindi or English.

2

The engine calculates their profile

In milliseconds, the app scores their risk tolerance (0–100), projects how much money they'll actually need after inflation, and filters all 9 investment instruments to find the best matches.

3

Shows a personalised investment plan

A clear breakdown: "Put 50% in Sukanya Samriddhi, 30% in PPF, 20% in Gold SIP" — with a reason for each choice, the expected return, and a direct link to open the account.

4

Partners (banks, NBFCs) pay to be listed

The B2B business model: financial institutions pay Niveshak.AI to be recommended to users. A partner form on the website captures their contact details for follow-up.

The Technology Stack

Think of building an app like building a house. React is the walls and rooms. TypeScript is the building inspector who checks everything is safe. Tailwind is the interior designer. Zustand is the memory (like post-it notes on a fridge). Supabase is the filing cabinet in the back office. Vercel is the delivery truck that puts the house on the internet.

Tool What It Is Why We Chose It
React 18 The UI framework — builds the buttons, screens, and components the user sees Industry standard, massive ecosystem, handles updates to the screen efficiently
TypeScript A stricter version of JavaScript that catches mistakes before the app runs Prevents bugs — if you pass the wrong data type, it tells you immediately
Vite 8 The build tool — bundles all the code files into files a browser can understand Extremely fast. In 0.7 seconds it compiles the entire app for production
Tailwind CSS A styling system — instead of writing CSS files, you add class names like bg-black text-lime Fast to write, consistent design, no style conflicts between components
React Router v6 Handles navigation — when you click a link, it shows the right screen without reloading the page The standard for React navigation
Zustand v4 State management — stores data (like your answers) that multiple screens need to share Simpler than Redux, tiny (1KB), persists data to localStorage automatically
i18n-js Internationalisation — switches all text between Hindi and English Simple API, works well with TypeScript

What is a Component?

A component is a reusable piece of the UI — like a LEGO brick. Instead of writing the same button 50 times, you write it once as a Button component and use it everywhere.

Example

The ScoreRing component draws the circular risk score chart. It takes a number (0–100) and draws the right arc. You write it once, use it on the results screen, share it in demos. Change the colour once in ScoreRing and it changes everywhere it appears.

The Brain
How the Engine Works

The engine is the most important part of Niveshak.AI. It is a series of pure functions — meaning they take input, do maths, and return output. No databases, no UI, no side effects. Just calculation.

⚙️

Why "pure functions"?

Because they are easy to test, debug, and reason about. Given the same input, you always get the same output. You can run the entire engine in a unit test with zero infrastructure.

Step 1 — Risk Scorer

File: src/engine/riskScorer.ts

This function reads 4 answers from the onboarding quiz and converts them into a score from 0 to 100:

AnswerMax PointsLogic
Investment experience25Never=0, FD only=5, Gold=12, Occasional MF=20, Regular=25
Risk tolerance35Panic sell=0, Worry but wait=10, Stay calm=25, Buy more=35
Timeline25Under 1yr=0, 1-3yr=5, 3-7yr=12, 7-15yr=20, 15+yr=25
Occupation15Daily wage=3, Retired=2, Student=10, Salaried=8-12, Self-employed=15

Then:

  • Score 0–33 → Conservative (safety first, government-backed instruments)
  • Score 34–66 → Balanced (mix of safe + market-linked)
  • Score 67–100 → Growth (equity-heavy, higher risk for higher return)

Step 2 — Inflation Projector

File: src/engine/inflationProjector.ts

If Arun says "I need ₹5 lakh for my daughter's education", that is the value today. In 10 years, education will cost more because of inflation. This function calculates the real target:

// Formula: Future Value = Present Value × (1 + inflation rate)^years
// Inflation rate assumed: 6% per year

₹5,00,000 × (1.06)^10 = ₹8,95,423

// So Arun actually needs ~₹9 lakh, not ₹5 lakh.
// This is what the "Corpus Target" card shows on the results screen.

Step 3 — Instrument Matcher

File: src/engine/instrumentMatcher.ts

This is the heart of the recommendation engine. It filters all 9 instruments through 4 gates:

GateQuestion AskedExample
TimelineIs the instrument's minimum timeline ≤ the user's timeline?SSY needs 15+ years. User has 10 years → SSY passes (10 ≥ 7-15 bucket)
RiskIs the instrument's max risk profile ≥ user's profile?ELSS is for Growth investors. Conservative user → ELSS fails
GoalDoes the instrument support this goal type?SSY supports Child Education + Child Marriage only
BudgetIs the instrument's minimum monthly ≤ user's surplus?NPS needs ₹500/month. User has ₹300 → NPS fails

After filtering, instruments are sorted (government-backed first for conservative users), top 4 are selected, and allocations are assigned:

ProfileAllocation Weights (top 4)
Conservative50% / 30% / 15% / 5%
Balanced40% / 30% / 20% / 10%
Growth40% / 25% / 20% / 15%

Complete Example: Arun's Journey

// Input: Arun's answers
{
  occupation:       "daily_wage",
  experience:       "never",
  riskTolerance:    "worry_and_wait",
  timeline:         "seven_to_15",
  inHandIncome:     2500,
  monthlySurplus:   800,
  goal:             "child_education",
  goalAmount:       500000
}

// Step 1: Risk Score
experience(0) + riskTolerance(10) + timeline(20) + occupation(3) = 33
→ Profile: CONSERVATIVE

// Step 2: Inflation Projection (10 years at 6%)
₹5,00,000 × 1.79 = ₹8,95,423

// Step 3: Filtered instruments for CONSERVATIVE + child_education + 10yr + ₹800

PPF   ✅ (min ₹500, govt, child education, 7-15yr)
NSC   ✅ (min ₹1000 — fails budget! ❌ if surplus < 1000)

// Step 4: Allocation (top 3, renormalized)

State Management
How the App Remembers

🧠 Analogy

Imagine the app is a conversation. You answer question 3. You move to question 4. The app needs to remember your answer to question 3. Without state management, every time you click "Next" the app forgets everything. State management is the app's memory.

What is Zustand?

Zustand (German for "state") is a library that creates a global store — like a shared whiteboard that any component in the app can read from or write to. It is much simpler than the older approach (Redux) and weighs only 1 kilobyte.

The Three Stores

1. useOnboardingStore — The Quiz Memory

Stores everything about the user's quiz progress:

{
  currentStep: 5,              // which question they're on
  answers: {
    name: "Arun",
    occupation: "daily_wage",
    goal: "child_education",
    goalAmount: 500000,
    timeline: "seven_to_15",
    ... and 6 more
  },
  isComplete: false            // has the quiz been finished?
}

2. useLanguageStore — The Language Memory

Just one value — which language is active:

{ language: "hi" }   // or "en"

3. useResultsStore — The Results Memory

Stores the computed recommendation so the results screen doesn't need to recalculate every time:

{
  result: {
    riskScore: 33,
    riskProfile: "CONSERVATIVE",
    recommendations: [SSY, PPF, RD],
    inflationProjection: { adjustedCorpus: 895423 }
  }
}

Why localStorage?

All three stores are persisted to localStorage — the browser's built-in key-value storage. This means:

  • If you close the browser tab and reopen it, your answers are still there
  • Your language preference is remembered across visits
  • The results page can be refreshed without losing data
🔍

Try it yourself

Open Chrome → F12 → Application tab → Local Storage → niveshak-onboarding. You can see your answers stored as JSON in real time as you fill the quiz.

React Hooks — The Connector Layer

Raw Zustand stores are like raw data. Hooks turn them into useful tools for the UI:

HookWhat it provides to UI components
useOnboarding() currentStep, currentStepData, goNext(), goPrev(), canContinue(), progress (0-100%)
useTranslation() t('key') function that returns Hindi or English based on active language
useRecommendation() Runs riskScorer → inflationProjector → instrumentMatcher, returns full RecommendationResult

Language System
Hindi + English

Why Bilingual Matters

English internet penetration in India is only about 30%. The remaining 70% of first-time investors are more comfortable in their regional language. Hindi reaches over 500 million people. Building bilingual from day one removes the single biggest barrier to adoption.

How It Works — The Data Layer

Every piece of user-facing text exists in both languages in the data files themselves. For example, in steps.ts:

{
  id: 'goal',
  questionEn: 'What do you want this money to do for you?',
  questionHi: '\u0906\u092A \u092F\u0939 \u092A\u0948\u0938\u093E \u0915\u093F\u0938 \u0915\u093E\u092E \u092E\u0947\u0902 \u0932\u0917\u093E\u0928\u093E \u091A\u093E\u0939\u0924\u0947 \u0939\u0948\u0902?',
  // Those \u codes are Unicode for Devanagari letters
  // In the browser they render as: आप यह पैसा किस काम में लगाना चाहते हैं?
}
⚠️ Important Technical Note

Hindi text is stored as \uXXXX Unicode escape codes (e.g. \u0906 for "आ") rather than the actual Hindi characters. This is because Windows PowerShell — which we used to write files — corrupts Devanagari characters when saving. Unicode escapes avoid this problem entirely. The browser converts them back to readable Hindi automatically.

The Language Toggle

The LanguageToggle component in the header is a simple button. When clicked, it calls useLanguageStore.setLanguage('hi') or 'en'. Because it's in Zustand (persisted to localStorage), the language choice survives page refreshes.

The Translation Hook

The useTranslation hook is how components access text:

const { language } = useLanguageStore();

// To show a question:
const question = language === 'hi'
  ? step.questionHi
  : step.questionEn;

// For an instrument name:
const name = language === 'hi'
  ? instrument.nameHi
  : instrument.nameEn;

The Database
Supabase Explained

🧠 Analogy

Supabase is like Google Sheets for your app, but much more powerful. Instead of a spreadsheet, it's a proper PostgreSQL database. Instead of sharing a link, your app talks to it via an API. Instead of everyone having full access, you set rules (RLS policies) for who can see what.

What We're Using Supabase For

Currently, only one thing: storing partner demo requests. When a bank fills the Partner form, that data goes to Supabase.

The partner_leads Table

create table public.partner_leads (
  id            uuid primary key default gen_random_uuid(),
  org_name      text not null,       -- "HDFC Bank"
  contact_name  text not null,       -- "Rajesh Agarwal"
  product_type  text not null,       -- "Mutual Funds"
  phone_email   text not null,       -- "rajesh@hdfc.com"
  created_at    timestamptz default now() -- auto timestamp
);

Row Level Security (RLS) — Why It Matters

RLS is a set of rules that controls who can do what with the data. We set up two rules:

Policy NameWhoWhat They Can DoWhy
anon_insert_leads Anyone (even without login) INSERT only The form on the website needs to save leads without user authentication
auth_select_leads Authenticated users only SELECT only Only you (logged into Supabase) can view the collected leads
🔐 Security Note

The anon key in the frontend code is safe to expose because RLS policies restrict it to INSERT only. Even if someone copies the anon key from the browser, they can only submit leads — they cannot read anyone else's data, delete records, or modify the table structure.

How the Form Saves Data

The flow from button click to database row:

1

User clicks "Request Demo"

DemoRequestForm.tsx calls submitDemoLead(formData)

2

lib/supabase.ts runs

Checks if Supabase is configured. Calls supabase.from('partner_leads').insert([...])

3

Supabase API receives the request

Validates the anon key, checks RLS policy, inserts the row

4

UI shows success or error

Form resets, button re-enables. You can now see the row in Supabase Table Editor.

Deployment
How the World Sees It

The Build Process

When you run npm run build, two things happen:

  1. TypeScript compiler checks every file for type errors. If anything is wrong, it stops here.
  2. Vite bundler takes all 349 modules and compresses them into a small set of optimised files in the dist/ folder.

The output is: 16 total files, ~415 KB total (after gzip compression: ~130 KB).

Code Splitting — Why the App Loads Fast

Instead of one giant JavaScript file, the app is split into chunks. The browser only downloads what it needs right now:

Vercel Deployment

Vercel is the hosting platform. Every time vercel --prod is run from the terminal, it:

  1. Uploads the source code to Vercel's servers in Washington DC
  2. Runs npm install and npm run build on their servers
  3. Distributes the dist/ folder to a global Content Delivery Network (CDN)
  4. Users in Mumbai get files from a Mumbai server. Users in Singapore from Singapore. Load time: <100ms.

The SPA Rewrite Rule

This is a subtlety about Single Page Applications. The app only has one HTML file (index.html). All routing (/onboard, /results) is handled by React Router in the browser.

But if a user pastes https://niveshak-web.vercel.app/results directly into their browser, the server would look for a file at /results, not find it, and return a 404. The fix in vercel.json:

// Redirect ALL URL paths to index.html
// React Router then reads the URL and shows the right screen
{
  "rewrites": [
    {
      "source": "/((?!assets|sw|manifest).*)",
      "destination": "/index.html"
    }
  ]
}

ViteThe tool that takes your 349 source files and compiles them into 16 optimised files a browser can understand. Very fast — takes 0.7 seconds for this project.

ZustandA tiny library (1KB) that creates a global state store — a shared memory that any component can read or write. Automatically saved to localStorage.
HookA React function that starts with "use" — like useOnboarding(). Hooks let components access shared logic without copy-pasting code.
localStorageA key-value store built into every browser. Data persists across page refreshes. Max size ~5MB. Used to save quiz answers and language preference.
SupabaseAn open-source Firebase alternative. Gives you a PostgreSQL database, API, authentication, and storage — all managed, no server setup required.
PostgreSQLOne of the world's most popular open-source relational databases. Supabase uses it under the hood.
RLS (Row Level Security)A database feature that lets you write rules like "users can only see their own rows". We use it so only authenticated users can read partner leads.
APIApplication Programming Interface — a way for two programs to talk to each other. Supabase provides a REST API so the app can read/write the database over HTTPS.
VercelA hosting platform optimised for frontend apps. One command (`vercel --prod`) deploys your app globally in under a minute.
CDNContent Delivery Network — a network of servers worldwide. Vercel serves your app from the server closest to the user, so load times are fast everywhere.
PWAProgressive Web App — a website that behaves like a native mobile app. Installable, offline-capable, full screen.
Service WorkerA background script (sw.js) the browser runs separately from the page. It intercepts network requests and serves cached files when offline.
WorkboxGoogle's library for writing service workers easily. We use CacheFirst strategy — serve from cache first, network only if cache misses.
Code SplittingBreaking the app's JavaScript into multiple files (chunks). The browser only downloads the chunk it needs — makes first load faster.
Lazy LoadingDelaying loading of a component until it's actually needed. The Results page code isn't downloaded until the user navigates to /results.
SPASingle Page Application — the entire app lives in one HTML file. Navigation between pages doesn't reload the page; React Router swaps the content instead.
i18nInternationalisation (abbreviated because there are 18 letters between the i and n). The practice of making software work in multiple languages.
Unicode / \uXXXXUnicode is the standard for encoding text in all world's languages. \u0906 is the code for "आ" in Devanagari. Used to safely store Hindi in English-encoded files.
GitVersion control — tracks every change made to the code. Like "track changes" in Word but for code. Every commit is a saved snapshot you can go back to.
GitHubA website that hosts git repositories (code history). Also the remote backup of the project.
Pure FunctionA function where the output depends only on the input. Given the same input, always returns the same output. No side effects. Easy to test.
env varsEnvironment variables — secret configuration values (like API keys) that live outside the code. Stored in .env.local file locally, in Vercel settings for production.
anon keySupabase's public API key — safe to expose in frontend code because RLS policies limit what it can do.
ELSSEquity Linked Savings Scheme — a type of mutual fund with a 3-year lock-in that qualifies for tax deduction under Section 80C.
SSYSukanya Samriddhi Yojana — government savings scheme for a girl child. 8.2% interest, tax-free. Minimum ₹250/month.
NPSNational Pension System — government-regulated retirement savings. Low cost, equity+debt mix, locked till age 60.
PPFPublic Provident Fund — 15-year government savings. 7.1% compound interest. Tax-free under EEE status.
